## 0. 一句话主张（放在 PPT 首页 & 邮件正文开头）

PromptUp 是一个 LLM-as-an-Optimizer 的 Prompt 自动优化系统：
- 自动生成/改写 Prompt（结构化模板 + 语义扩展 + 关键词增强 + 搜索空间组合）
- 自动调用 LLM（NVIDIA / OpenAI）执行任务
- 自动评估指标（Accuracy / ROUGE / BLEU）
- 自动用搜索算法（随机 / 贝叶斯 / 遗传）迭代找到最优 Prompt
- UI 中直接输出最优 Prompt + 得分曲线 + 搜索过程可视化，并提供关键词贡献度分析（加分项）

## 2. 赛题要求 → 你们系统对应功能（PPT 第二页可直接用）

- 1 自动生成 Prompt
	- 生成/分类/摘要/翻译的 Meta-Prompt 模板与结构化输出
	- 搜索空间生成（角色/风格/技巧组合）

- 2 自动调用大模型 API
	- 支持 NVIDIA 与 OpenAI 两种 Provider，可在侧边栏切换

- 3 自动计算表现指标
	- 分类：Accuracy
	- 摘要：ROUGE-1/2/L（支持中文分词）
	- 翻译：BLEU（支持中英文分词/平滑）

- 4 优化算法寻找最优 Prompt
	- 随机搜索、贝叶斯优化、遗传算法
	- UI 内置“提示词优化实验室”可直接跑

- 5 输出最优 Prompt、得分变化曲线与可视化
	- UI 输出最佳 Prompt、组合（role/style/technique）、搜索空间预览
	- 随机搜索/贝叶斯/遗传都有得分曲线（线图）

加分项覆盖：
- 多任务优化：四任务完整支持
- 关键词贡献度分析：UI 自动分析并可下载 CSV
- 图形化界面：Streamlit 现代化 UI

### A) 5 分钟高分版（只演示最强链路：分类 + 搜索算法曲线）

目标：在最短时间内让评委看到“自动评估 + 自动搜索 + 曲线收敛 + 最优 Prompt 输出”。

2. 侧边栏：选择 分类任务 + NVIDIA + 模型（推荐 meta/llama-3.3-70b-instruct 或更强）

1. 找到“🔬 效果验证实验室”

3. 点击“🧬 运行遗传算法”
4. 展示：
	 - 进度条（每代最佳/平均分）
	 - 进化曲线（最佳得分、平均得分随代数变化）

4:30–5:00 收尾（强调可扩展与可信）
- 同一套框架也支持摘要/翻译（ROUGE/BLEU），并提供可复现证据：静态编译检查 + API 连通脚本 + 算法对比图。

## 5. 演示时“说什么更加分”（推荐话术要点）

1) 为什么需要优化
- Prompt 决定任务约束、输出格式、推理路径；人工调参慢且不可复现。

2) 我们的核心闭环
- 生成候选 Prompt → 让模型执行任务 → 指标打分 → 算法选择下一批候选 → 输出最优 Prompt + 曲线。

3) 为什么是三类算法
- 随机搜索：快速 baseline
- 贝叶斯优化：评估昂贵时的“样本效率”
- 遗传算法：离散组合空间里稳定进化、曲线直观

4) 为什么关键词贡献度分析加分
- 不只输出结果，还能解释“哪些词/约束对 Prompt 贡献最大”，增强可解释性与可迁移性。

## 6. PPT 推荐结构（12 页左右，信息密度高且不水）

下面这一节请你“整段复制”给做 PPT 的 AI（如 Gamma / Tome / PPT Copilot 等）。目标是：AI 在没看过源代码的情况下，也能按描述生成可交付的 PPT。

### 6.0 全局排版规范（PPT AI 必须严格遵守）

PPT 画布与风格：
- 画布：16:9（宽屏）
- 语言：中文为主，必要英文术语括号补充（如 ROUGE-L）
- 字体：标题「微软雅黑」或「思源黑体」；正文字体同一套；代码/命令使用等宽字体（Consolas）
- 字号建议：
	- 页标题 36–44
	- 二级小标题 24–28
	- 正文要点 18–22
	- 备注/脚注 12–14
- 网格：建议 12 栏栅格；页面四边安全边距 ≥ 5%
- 统一页眉区：顶部 12% 高度放“页标题 + 1 句结论（Key Takeaway）”；主体内容占 88%
- 每页最多 6 条要点（强制），每条 ≤ 18 个字；超出就拆页或改为图表
- 颜色：使用“深色文字 + 浅色底”简洁商务风；强调色只用 1 种（建议深蓝），不要花里胡哨
- 图标：仅用于辅助识别（任务类型/模块），保持风格一致（线性图标）

图片占位规则（你后续补截图）：
- 占位图统一用浅灰色矩形块，写清：图片来源、需要截图的页面区域、建议截图尺寸比例
- 每张占位图下方用小字写：文件名建议（例如 cls_ga_curve.png）

页面编号规则：
- 每页右下角用小字显示“PromptUp | 学院程序设计大赛 | 页码/总页数”

讲稿备注规则（Speaker Notes）：
- 每页底部附 2–4 条“讲什么”，用于答辩/录屏口播

页标题：为什么需要 Prompt 自动优化？

Key Takeaway：Prompt 调参本质是“离散组合 + 不确定反馈”的优化问题，必须自动化才能可复现。

右栏（我们对齐赛题的目标，3–4 条）：
- 自动生成候选 Prompt
- 自动调用 LLM 执行任务
- 自动计算指标（Accuracy/ROUGE/BLEU）
- 自动搜索最优 Prompt + 曲线可视化

视觉元素：
- 右下角放一个“闭环小图”（四步箭头环）占位即可：生成→执行→评估→优化

## 11. 答辩 FAQ：算法效果不佳 & 论文看不懂怎么办

这一节专门用来“兜底”：即使三种算法最后没把分数提很高，你也能自信讲清楚你们完成了赛题的工程闭环，并且对效果不佳的原因与改进路径有清晰判断。

### 11.1 如果三种算法优化效果不明显，怎么办？（推荐答辩话术）

先给一个“可信解释框架”，不要硬吹：
- 我们实现的是可复现的“搜索-评估-迭代”闭环，效果由三件事共同决定：搜索空间质量、评估数据与指标噪声、模型本身可控性。
- 在小样本/高噪声评估下，算法之间的差异会被噪声淹没；但系统能稳定输出曲线、最佳组合与可复制 Prompt，这就是工程化价值。

你可以直接说这段（30 秒版）：
- “我们把 Prompt 优化问题工程化成闭环：候选生成→执行→指标打分→算法迭代。当前实验里提升不大，主要是小样本评估噪声和搜索空间上限导致，但三种算法能稳定跑通、输出最优 Prompt 和曲线，并且我们清楚如何通过扩大数据、改进目标函数与搜索空间来提升效果。”

评委追问“为什么不提升？”你给 3 个常见原因（按优先级）：
1) 评估集太小：5-10 条样本时，Accuracy/ROUGE/BLEU 方差很大
2) 搜索空间太窄：role/style/technique 组合如果相差不大，分数天然拉不开
3) 目标函数噪声：同一 Prompt 多次调用 LLM 可能输出波动（温度、随机性）

你立刻给 3 个“现场可做的改进动作”（不需要读论文）：
1) 固定评估：把 temperature 调低或设为 0（减少随机）
2) 增大评估集：默认数据从 10 条扩到 30 条（哪怕是复制扩充 + 人工补几条边界样本）
3) 多次评估取均值：同一候选 Prompt 跑 2-3 次取平均分（成本↑但更稳）

展示策略（非常关键）：
- 不强调“绝对分数提升多少”，强调“曲线形态 + 稳定性 + 可解释输出”。
- 用“算法对比图”讲差异：随机波动 vs 遗传逐代改进 vs 贝叶斯样本效率。

### 11.2 怎么把“效果一般”讲成加分点（评委更吃这一套）

你把亮点从“结果数值”转移到“工程能力与科学实验方法”：
- 我们没有只做一个 demo Prompt，而是把问题形式化为优化问题，并实现三种算法可插拔对比。
- 我们提供可量化指标与可视化曲线，能复现实验过程，而不是“玄学调 Prompt”。
- 我们还做了关键词贡献度分析，能解释 Prompt 的关键约束来源。

### 11.3 看不懂顶会论文怎么办？（三层阅读法 + 落地路径）

你完全不需要把论文全读懂才能做出“比赛加分”的系统。推荐三层阅读法：

第一层（15 分钟，拿结论）：
- 只看 Abstract / Introduction / Conclusion
- 只回答 3 个问题：
	1) 论文优化的对象是什么（prompt/策略/模型参数）？
	2) 它的反馈信号是什么（指标/人类偏好/打分器）？
	3) 它声称解决的核心痛点是什么（样本效率/稳定性/泛化）？

第二层（30-60 分钟，拿方法骨架）：
- 只看 Method 图和 Algorithm 伪代码
- 把复杂方法“翻译”为你能实现的 4 步模板：
	1) 候选怎么产生
	2) 怎么评估

第三层（可选，按需深挖）：

- 你可以把论文方法抽象为：更好的搜索空间、更稳定的评估、更聪明的采样（这三点你们系统都能承接）。

### 11.4 如果评委问“你们算法是不是 AI 写的？”怎么回答

### 11.5 评委问：为什么“贝叶斯优化”能用在 Prompt 优化里？它内部为什么有效？（标准回答）

你先给“适用性一句话”（10 秒）：
- “因为我们要优化的是一个黑盒函数：给定 Prompt 组合 → 调一次大模型 → 得到一个分数。每次评估都贵且有噪声，贝叶斯优化专门用于‘昂贵黑盒函数’的样本高效优化。”

再给“数学直觉两句话”（20–30 秒，够用了）：
- “贝叶斯优化把未知目标函数 f(x) 当成随机变量，用历史数据拟合一个‘代理模型/概率模型’，从而得到‘哪里可能更高分’的不确定性估计。”
- “它用采集函数（比如 Expected Improvement）在‘探索不确定区域’和‘利用当前高分区域’之间做折中，所以通常比纯随机更省试验次数。”

如果评委追问“你们这里用的 TPE 到底在做什么？”（1 分钟展开版）：
- 我们的参数 x 是离散类别变量：x = (role, style, technique)。每次试验得到一个观测 y = f(x)（指标分数）。
- Optuna 的 TPE（Tree-structured Parzen Estimator）不是直接拟合 p(y|x)，而是拟合两个密度：
	- l(x) = p(x | y \le y*)（“好结果”区域的参数分布）
	- g(x) = p(x | y > y*)（“一般/差结果”区域的参数分布）
- 然后它倾向于选让比值 l(x)/g(x) 更大的 x（直观上：这个组合更像“高分组”，更不像“低分组”），等价于在近似最大化期望改进（EI）的一种方式。

你可以把它讲得更“工程化”一点（评委更好接受）：
- “随机搜索每次瞎抽；TPE 会‘记住’哪些 role/style/technique 在高分试验里更常出现，然后更频繁地往这些区域试，同时仍保留部分探索避免局部最优。”

### 11.6 评委问：那为什么你们这里贝叶斯优化提升不明显？（把劣势说成专业判断）

你用“不是算法不行，是实验条件限制”来解释（30 秒版）：
- “贝叶斯优化优势在‘每次评估昂贵、试验次数很少’时体现。但我们这里的搜索空间是离散组合，而且评估噪声比较大（LLM 随机性/小样本），再加上试验次数可能偏少，导致模型学到的‘高分规律’不稳定，收益就不明显。”

展开说清楚 4 个技术原因（按常见程度排序）：
1) 目标函数噪声大：同一 x 多次调用 LLM，y 会抖动；噪声会误导 TPE 对 l(x)/g(x) 的估计
2) 样本太少：n_trials 只有 8–12 时，还处在“startup 随机探索”阶段，TPE 来不及形成稳定偏好
3) 搜索空间太小/太浅：roles×styles×techniques 的组合数不大时，随机本来就能覆盖较多区域，贝叶斯优势变小
4) 离散且非平滑：prompt 组合的得分可能呈“跳变”，不是连续平滑函数，代理模型的收益下降

你再给“我们知道怎么让它更有效”的补救点（不改论文也能做）：
- 固定随机性：temperature 设低或 0；同一组合评估 2–3 次取均值
- 增大测试集：把评估样本从 10 扩到 30（方差下降）
- 增加试验次数：n_trials 提到 20–30（让 TPE 真正进入‘学习后利用’阶段）
- 工程化去重：如果出现重复组合，做一次缓存/去重（避免浪费试验预算）
